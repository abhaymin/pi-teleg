Primary intent

Turn private chats, groups, supergroups, and configured channels into a remote control plane for multiple Pi sessions, while using SQLite to coordinate queue ownership, session liveness, bot scoping, and recovery.

Intent

teleg-bridge connects Telegram chats to Pi agent sessions. It receives Telegram messages, persists work in SQLite, routes each message to the best live session, and sends text, photos, videos, or generated files back to Telegram.

Remote control plane

Message Pi from Telegram and receive the final answer or attached artifacts.

Capability routing

Use @sessionName, INFO_REL.md, URL patterns, and primary fallback.

Recoverable queue

SQLite preserves pending, processing, completed, and failed work across crashes.

SQLite database architecture

SQLite is the shared coordination layer. It is local-first, durable, and safe for multiple Pi sessions through WAL mode and bot-scoped queries.

Path

Default: ~/.pi/agent/teleg-bridge.db. Override with TELEG_DB_PATH.

Runtime pragmas

journal_mode=WAL, busy_timeout=5000, synchronous=NORMAL.

Scope

bot_id scopes messages, sessions, downloads, and pub-sub rows.

Workflow

Setup

Install, build, deploy, then configure the bot with /teleg-setup, config files, or environment variables.

Session registration

Resolve BotContext, detect capabilities from INFO_REL.md, start relay server, register in SQLite, and heartbeat.

Polling

One session per bot acquires polling-{botId}.lock; other linked sessions remain available for routed work.

Routing

Commands are handled first, then direct route, capability route, and primary fallback.

Response

The selected session completes the [telegram] turn and the bridge sends replies or attachments.

Reconciliation

Reconcile sessions, evict ghosts, reset stuck queue rows, and elect a primary.

Routing and capabilities

Sessions declare capabilities in INFO_REL.md. The router checks direct targets and capability matches only if sessions are live.

# INFO_REL

## capabilities
telegram-bridge, messaging, relay, routing

## description
Multi-session Telegram bridge extension for Pi. Handles polling, session routing, message relay, and capability-based smart message distribution between Pi sessions.
1 Telegram commands 2 @sessionName 3 capability match 4 primary fallback

Telegram I/O

Polling private chats, groups, supergroups, and configured channels; authorized intake, chunked replies, media sends, attachments, typing indicator.

Multi-session routing

Direct routing, capability matching, primary fallback, cross-session HTTP relay.

SQLite persistence

Message queue, download queue, relay registry, relay history, and pub-sub delegation.

Reliability

WAL mode, per-bot locks, stale message recovery, liveness checks, ghost eviction.

Group and channel authorization

Privatefrom.id must be in allowedUserIds, except first pairing.
Group / supergroupThe configured user can request from that chat; replies go back to that group message.
ChannelChannel posts normally have no user ID, so chat.id must be in allowedChatIds.

In groups, BotFather privacy mode controls whether the bot receives all messages or only commands, replies, and mentions. Group commands like /status@YourBot are normalized to /status.

Operational guardrails

Do not split DBs for one bot

Sessions sharing a bot should share the same TELEG_DB_PATH.

Evict ghost sessions

Use /teleg-reconcile or teleg-evict_session after hard kills.

Declare capabilities

Add INFO_REL.md so Telegram messages route to the correct workspace.
